# Vib+ Build Roadmap (In-House Language + SDL Graphics)

## Phase 0: Repository bootstrap (1 week)

- Define stable grammar subset for v0.
- Lock stdlib surface needed by existing game examples.
- Create test corpus (`valid/invalid` source files).

## Phase 1: Console-first compiler (3-5 weeks)

- Lexer: tokens, spans, diagnostics.
- Parser: functions, variables, records, enums, loops, match.
- Type checker: primitives, function signatures, basic inference.
- C backend (temporary) generated by us for fast bring-up.
- Goal: `vibc run hello.vp` and simple CLI tools.

### Progress snapshot

- Lexer/parser bootstrap is implemented.
- `vibc check`, `tokens`, `ast`, and `ir` are available.
- `vibc build` / `run` are wired for game entry targets through SDL runtime mapping.

## Phase 2: Runtime and stdlib core (2-4 weeks)

- Runtime startup + allocator.
- `std/console`: print/readline.
- `std/io` and `std/fs`: file read/write.
- Error model (`Result`) and process exit codes.
- Goal: compile and run console applications end-to-end.

## Phase 3: Native backend (4-8 weeks)

- Vib IR design and lowering from typed AST.
- x86-64 codegen for arithmetic, branches, calls.
- Windows object/PE emission and linking pipeline.
- Goal: remove temporary C backend for core programs.

## Phase 4: Graphics subsystem with SDL2 (2-4 weeks)

- SDL2 window creation and event loop.
- SDL2 renderer abstraction for rect/line/circle/text.
- Input subsystem (keyboard/mouse states by frame).
- `std/game` implementation backed by SDL2.
- Goal: run Pong and Flappy-style game.

## Phase 5: Chess and tooling hardening (2-4 weeks)

- Optimize draw/update loop and frame timing.
- Add unit tests and snapshot tests for compiler passes.
- Add formatter and package scaffolding (`vpm` minimal).
- Goal: run all 3 games and console examples from one CLI.

## Initial deliverables checklist

- `vibc build <file.vp>`
- `vibc run <file.vp>`
- `vibc check <file.vp>`
- `vibc test`
- `std/console` and `std/game` minimum API

## Risks and controls

- Backend complexity risk -> keep temporary C backend while native backend matures.
- Debugging complexity -> source span metadata retained through all passes.
- Graphics API churn risk -> keep SDL details isolated in `runtime/platform/sdl/*`.
